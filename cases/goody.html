Goody - 팬 굿즈 거래 앱 MVP

팬 굿즈 거래는 상품 등록만으로 끝나지 않습니다. 사용자는 판매자와 대화하고, 거래 상태를 확인하고, 완료 이후 후기를 남겨야 합니다. Goody 사례는 거래형 앱에서 필요한 핵심 흐름을 MVP 범위로 정리한 포트폴리오 예시입니다.

유형
모바일 앱 / 거래 서비스
범위
앱 화면, API, 거래 상태, 알림
기간 예시
6~8주 MVP
핵심 가치
거래 흐름과 신뢰 장치 설계

문제 정의

굿즈 거래 서비스의 핵심 문제는 사용자가 상품을 발견한 뒤 거래를 완료하기까지 여러 행동이 끊어진다는 점입니다. 상품 등록, 채팅, 거래 확정, 후기 작성, 알림이 따로 움직이면 사용자는 현재 상태를 놓치고 운영자는 분쟁 상황을 파악하기 어렵습니다.

따라서 MVP에서는 모든 기능을 한 번에 크게 만드는 대신, 거래가 시작되고 완료되는 최소 경로를 안정적으로 만드는 것을 우선순위로 잡았습니다.

접근 방식

01거래 상태 모델링

등록, 문의, 진행, 완료, 취소 상태를 나누고 각 상태에서 가능한 사용자 행동을 정리했습니다.

02상품 상세 중심 플로우

상품 상세에서 채팅, 찜, 거래 요청, 상태 확인으로 이어지는 흐름을 구성했습니다.

03신뢰 장치 배치

후기, 신고, 거래 히스토리처럼 사용자가 상대를 판단할 수 있는 정보를 우선 노출했습니다.

구현 범위

  • 상품 등록, 수정, 삭제, 이미지 업로드 흐름
  • 상품 목록, 상세, 검색, 카테고리 필터
  • 판매자와 구매자 간 채팅 진입 및 거래 문의 흐름
  • 거래 상태 변경과 완료 후 후기 작성 시나리오
  • 푸시 알림 또는 알림 목록으로 주요 이벤트 전달
  • 관리자가 신고, 상품, 회원 상태를 확인할 수 있는 기본 운영 구조

산출물

결과물은 앱 화면만이 아니라 서비스 운영에 필요한 기준까지 포함해 정리합니다. 화면 흐름도, API 기능 목록, 데이터 구조, 검수 체크리스트를 함께 두면 출시 전 빠진 기능을 확인하기 쉽습니다.

사용자 앱목록, 상세, 등록, 채팅, 후기
서버/API회원, 상품, 거래, 알림 데이터
운영 기준신고, 거래 상태, 검수 체크리스트

핵심 포인트

  • 거래 서비스는 화면보다 상태 설계가 중요합니다.
  • MVP에서는 결제보다 탐색, 문의, 신뢰 흐름을 먼저 검증합니다.
  • 운영자가 확인할 데이터 구조를 초기에 잡아야 이후 확장이 쉽습니다.
